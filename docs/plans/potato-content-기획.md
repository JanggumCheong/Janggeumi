# 감자 콘텐츠 강화 기획

> 대상: `potato.json` (이미 `published`). 이번 작업은 **새 재료 추가가 아니라, 이미 얕게 담긴 답에 "왜"를 채우는 콘텐츠 보강**이다.

## 1. 한 줄 정의

감자 상세의 **껍질 처리 · 싹 · 냉장/상온** 세 질문에 "결론"뿐 아니라 **"왜 그런가"의 검증된 근거**를 붙여, 검색·AI가 인용할 수 있는 콘텐츠로 만든다.

## 2. 문제 (Why)

감자는 이미 published지만, 사용자가 실제로 검색하는 질문들이 데이터에 **한 줄 결론으로만** 담겨 있다:

| 사용자 질문 | 현재 potato.json | 빈 곳 |
|---|---|---|
| 감자 껍질 일반? 음식물? | `dispose.peel.way` 한 줄 = 음식물 | **왜 음식물인지·지역차·출처 없음** |
| 감자 싹 나면 어떻게 / 왜 나나 | `dispose.sprout` 한 줄 + 빈 `related` 링크 | **"왜 싹이 나나" 콘텐츠 자체가 없음** |
| 냉장? 상온? | storage `tip`·`cautions`에 "냉장 해로움" | **"냉장이 왜 나쁜가" 정면 설명 부재**, 냉장 필터 옵션에 매칭 카드 없음 |

"결론만 있고 왜가 없으면" 사용자는 확신하지 못하고(빠른 확신 실패), AI는 인용하지 못한다(GEO 실패). 이 서비스의 핵심은 **검증된 근거 위에 올린다**는 것인데, 그 근거가 비어 있다.

## 3. 대상 & 시나리오 (Who/When)

- **대상:** 감자 한 봉지를 사서 며칠 지나 마주치는 자취·1인 가구. "이거 버려야 하나?", "싹 났는데 먹어도 되나?", "냉장고 넣을까?"의 순간.
- **시나리오 A (껍질):** 감자전 해먹고 껍질이 한 움큼. "일반이지? 아니 음식물인가?" → 감자 상세 처리탭 폐기 세그먼트에서 **"음식물 (단, 지역 확인)"** + 왜인지(사료화 가능) 확인 → 확신하고 버림.
- **시나리오 B (싹):** 사둔 감자에 싹이 남. "왜 났지, 먹어도 되나?" → 처리탭에서 **싹 난 감자 처리** + related "감자 싹, 왜 나고 먹어도 될까?"로 이동 → 도려내면 되는지/버려야 하는지 판단.
- **시나리오 C (냉장):** 여름이라 냉장고에 넣을까 고민 → 보관탭에서 **"냉장은 오히려 해로워요"** + 저온당화·아크릴아마이드 근거 확인 → 냉암소로.

## 4. 핵심 가치 매핑

- **빠른 확신** — 세 질문 모두 "결론 + 왜"를 한 화면에서. 껍질=음식물 배지, 냉장=❌ 명시. 망설임 제거.
- **믿음(집단지성/권위)** — 결론마다 **출처**(식약처·환경부·농진청)를 붙인다. 특히 분리배출·독성은 임의로 지어내지 않고 공식 기준만. 이 재료는 **구매·보관=정답형**이라 권위가 무기.
- **끝까지(안 버리기)** — 싹·초록·껍질을 "무조건 버려"가 아니라 **도려내면 먹을 수 있다 / 껍질은 말려 활용** 등 안 버리는 길을 먼저 제시.

## 5. 영역 & 정보 성격

세 질문이 **세 영역에 각각** 걸린다 — 감자 상세 3탭 전체를 건드린다:

| 질문 | 영역 | 정보 성격 | 반영 위치 |
|---|---|---|---|
| 냉장/상온 | 🧊 보관 (합의형) | 필터+비교. 단 "냉장 나쁨"은 정답에 가까움 | `storage.methods`에 **냉장 카드 추가**(비추천으로), `tip`·`cautions` 근거화 |
| 싹 (왜/처리) | 🍳 처리 (확산형) + 🛒 구매 | 싹 원인=정답형 지식, 처리=폐기 안내 | `handling.dispose.sprout` 상세화 + `related` 실콘텐츠, 구매 `no-sprout` 근거 유지 |
| 껍질 | 🍳 처리 (폐기=정답형) | 지자체 기준 = 정답형 | `handling.dispose.peel` reason/regionNote/source 채움 |

## 6. 기능 명세 (화면·동작)

각 항목 = 사용자 행동 → 시스템 반응. **화면은 새로 만들지 않는다** — 기존 3탭 목업(`ingredient-detail-mockup.html`)의 폐기 카드·보관 카드·related가 이미 이 데이터를 렌더한다. 채우는 건 데이터.

### 6-1. 보관탭 — "냉장? 상온?" 정면 답

- 사용자가 보관탭 진입 → **냉암소 실온 카드(추천, 1위)** 가 상단. 기존 `dark-cool-apple` 유지.
- **냉장 카드 신규 추가**(`recommended:false`, 필터 `place:냉장`에 매칭 → 깨진 필터 정합성 복구). 카드 요약 = "냉장은 오히려 해로워요". `cautions`에 **저온당화 → 아크릴아마이드** 근거 한 줄. 출처 식약처.
  - → 사용자가 `place=냉장` 필터를 눌러도 빈 결과가 아니라 "왜 안 되는지" 카드가 뜬다. **필터가 곧 설명이 된다.**
- 하단 `tip` = "감자의 3대 적: 빛·습기·냉장" 유지, 근거 링크.

### 6-2. 처리탭(폐기) — "껍질 어디에 버려요?"

- 폐기 세그먼트 → `peel` 카드: `way`="음식물로 배출", `wasteType` 배지 = **음식물**.
- 카드 탭 → 상세: **`reason`**(수분 많아 사료·퇴비화 가능 → 음식물) + **`regionNote`**(지자체 처리시설 따라 다름, 관할 확인) + **`source`**(식약처 분리배출 안내).
- **대비 포인트:** 같은 감자라도 껍질=음식물이지만, 참고로 "양파·마늘 마른 껍질은 일반쓰레기(사료 부적합)"라는 원칙을 `reason`에 녹여 판별 기준("동물이 먹을 수 있나")을 학습시킨다.

### 6-3. 처리탭(폐기) — "싹 난 감자"

- `sprout` 카드: `way`="싹·초록 부분 도려내고, 많이 나면 음식물로", `wasteType`=음식물.
- 상세 `reason` = **솔라닌·차코닌 독소**가 싹·초록에 집중. 조금이면 도려내고 먹기 가능, 많으면 폐기.
- `cautions` = "도려낸 뒤에도 쓴맛이 나면 먹지 말 것".

### 6-4. related — "감자 싹, 왜 나고 먹어도 될까?"

- 기존 빈 링크(`/재료/potato/싹`)를 **실제 지식 콘텐츠**로 승격. h1 = 검색어형 "감자 싹, 왜 날까? 먹어도 되나요?".
- 내용: **싹 나는 이유**(감자는 살아있는 덩이줄기 → 빛·온도·습도가 발아 촉진) + 솔라닌 + 안전 손질. 이게 **SEO/GEO 진입 콘텐츠**.

## 7. 데이터 & 필터

### 채울 필드 (potato.json)

```
storage.methods[] += {                       // 냉장 카드 신규
  id: "fridge-avoid", recommended: false,
  tags: { place: "냉장", light: "일반", companion: "단독" },  // ← place:냉장 필터 정합성 복구
  summary: "냉장은 오히려 해로워요",
  cautions: ["저온에서 전분이 당으로 변해(저온당화)...",
             "그 감자를 고온에 튀기면 아크릴아마이드↑"],
  source: 식약처
}

handling.dispose.peel += {
  reason: "수분 많아 사료·퇴비화 가능 → 음식물. (양파·마늘 마른껍질은 사료 부적합→일반)",
  regionNote: "지자체 음식물 처리시설에 따라 기준이 달라요. 애매하면 관할 확인.",
  source: { org:"식품의약품안전처", type:"official", url:"https://www.mfds.go.kr", ... }
}

handling.dispose.sprout += {
  reason: "싹·초록 부분에 솔라닌·차코닌 독소 집중. 조금이면 도려내고, 많으면 폐기.",
  cautions: ["도려낸 뒤에도 쓴맛이 나면 먹지 말 것"],
  source: { org:"식품의약품안전처", ... }
}

related[0] → 실제 콘텐츠로 (h1: "감자 싹, 왜 날까? 먹어도 되나요?")
```

### 필터 정합성 (핵심)

- 현재 `storage.filters.place` 옵션 = `["냉암소(실온)", "냉장", "냉동"]` 3개인데, 방법은 냉암소·냉동 2개뿐 → **냉장 옵션이 빈 필터**. 냉장 카드 추가로 **3옵션 : 3카드 정합** 복구.
- 새 필터 축은 추가하지 않는다(3축 유지: place·light·companion). schema의 "최대 3축" 준수.

### SEO / GEO

- **보관탭 h1:** "감자 보관 방법 비교" (`storage.headline`, 유지).
- **싹 콘텐츠 h1:** "감자 싹, 왜 날까? 먹어도 되나요?" — 검색어 그대로.
- **JSON-LD:** 폐기·싹 결론에 `source`(식약처·환경부) → `HowTo`/`FAQPage` 구조화 데이터 근거. 출처 접힘 영역도 **SSR로 DOM에 존재**시켜 색인.
- **장금이 톤:** 냉장 카드 코멘트 = "👩‍🍳 장금이 왈, 감자는 냉장고에 넣지 마시옵소서 — 오히려 해로운 것이옵니다."

## 8. 성공 지표 (How measure)

- "감자 껍질 음식물", "감자 싹 왜", "감자 냉장 보관" 검색 유입 → 감자 상세/싹 콘텐츠 랜딩 수.
- 폐기 카드 → 상세 전개(탭) 비율(=근거를 실제로 읽는가).
- AI 검색(Perplexity·AI Overviews)에서 감자 분리배출·싹 답변에 **출처로 인용**되는지(GEO).
- 보관탭 `place=냉장` 필터 클릭 후 이탈률 하락(빈 결과 → "왜 안 되는지" 카드로 전환).

## 9. 범위 밖 / 미정

- **범위 밖:** 새 화면·컴포넌트 개발(기존 3탭 목업 재사용). 이미지 에셋(`image:null` 유지). 레시피 UGC 추가.
- **미정 (팀 확인):**
  - 싹 콘텐츠(`related`)를 **별도 라우트 페이지**로 만들지, 처리탭 안 확장 블록으로 둘지 → 지금 기획은 별도 라우트(`/ingredients/potato/sprout`) 가정. SEO 독립 URL이 유리.
  - 분리배출 `source`의 정확한 딥링크 URL — 식약처 정책브리핑(2021, 정책브리핑 korea.kr)·환경부 "내 손안의 분리배출" 중 어느 것을 정식 출처로 고정할지. 지금은 기관명·type만 확정, URL은 대표 URL.
  - 냉장 카드의 `rating` — 비추천 카드에 별점을 붙일지(사용자 후기는 후속 UGC). 지금은 **장금이 코멘트로 대체**, rating 생략 가능.
