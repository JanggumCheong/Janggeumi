# 없는 재료 LLM 발화 안내 — 기획

## 1. 한 줄 정의

정식 데이터가 없는 재료(없는 상세 페이지·검색 무결과)에서, `POST /api/ai/chat`로 **장금이가 즉석 발화로** 구매·보관·손질을 임시 안내하는 기능.

## 2. 문제 (Why)

- 정식 콘텐츠는 재료가 **6개뿐**(딸기·사과·샤인머스캣·수박·아보카도·천도복숭아). 그 외 재료(당근·양파·고구마…)는 진입 시 **404** 또는 검색 **무결과** → "끝까지 안 버리기"(빈손 방지) 가치에 어긋남.
- 사용자는 "지금 이 재료" 정보를 원하는데, 데이터가 아직 없다고 빈 화면을 주면 이탈한다.
- **대안:** LLM이 장금이 톤으로 즉석 안내 → 데이터 공백을 발화로 메운다. "정식 콘텐츠 준비 전까지의 임시 다리".

## 3. 대상 & 시나리오 (Who/When)

- **당근 검색:** 검색창에 "당근" → 매칭 재료 없음 → "조건에 맞는 재료가 없어요" 대신 **장금이가 당근 안내를 발화**.
- **없는 재료 진입:** `/ingredients/carrot` → 정식 데이터 없음(404 대신) → 장금이 발화 카드.
- **이상 입력:** 검색에 "asdf" → API가 이미 되묻기("어떤 재료가 궁금하신가요?") → 그대로 노출.

## 4. 핵심 가치 매핑

- **끝까지(안 버리기):** 데이터 없다고 빈손으로 돌려보내지 않는다. 발화로라도 안내 = 핵심 지지.
- **빠른 확신:** 즉석 답이라도 구매/보관/손질 구조로 바로 확신을 준다.
- **믿음(집단지성):** ⚠️ 여기선 **약함** — LLM 생성이라 검증 안 됨. 그래서 **"임시/AI 생성" 명시**로 정식 콘텐츠(검증됨)와 구분해 신뢰 모델을 지킨다.

## 5. 영역 & 정보 성격

- 이 기능은 **특정 영역(구매/보관/처리)이 아니라 "데이터 공백 대체"** 성격.
- 단 answer가 **구매/보관/손질** 섹션으로 오므로, 정식 3탭(구매·보관·처리)의 **축소판**으로 파싱해 보여준다. (손질 ≈ 처리)
- 정식 화면과 **명확히 구분** — 필터·평점·출처 없음. 장금이 발화 카드 하나.

## 6. 기능 명세

### 6-0. 말투 — 장금이 톤 (가볍게)
- 정식 3탭 화면처럼 **"장금이 왈, ~것이옵니다"** 톤을 쓰되 **과하지 않게**. 자연스러움이 우선.
- **인트로·마무리만** 가볍게 장금이 톤(예: "제가 아는 대로 일러드리는 것이옵니다"). **정보 리스트는 평이한 존댓말**("~좋아요 / ~하세요")로 — 지금 `ai/chat` answer 톤과 그대로 맞는다.
- "~하시옵소서 / ~다스리옵소서" 같은 **과한 사극체 금지.** 읽기 부담·이질감만 준다.
- 즉 **API answer(친근체)를 거의 그대로** 쓰고, 카드 인트로/푸터의 프론트 카피에만 살짝 장금이 색을 입힌다. (별도 톤 변환 불필요)

### 6-1. 화면 — 장금이 메시지 카드
- 사용자 행동: 없는 재료 진입 / 검색 무결과 → 시스템: `ai/chat` 호출 → **장금이 발화 카드** 렌더.
- 카드 구성:
  - 👩‍🍳 **장금이 화자** + "임시 안내" 배지 (AI 생성 명시)
  - answer의 **인트로 문장** → 카드 상단 카피
  - **구매 / 보관 / 손질** 섹션 파싱 → 각 접이식/구분 블록 (`- ` 리스트)
  - 마무리 문장 → 하단 보조 문구
  - "정식 콘텐츠 준비 중" 안내 + (선택) 정식 재료 추천 링크
- **파싱 규칙:** answer 텍스트를 `구매`/`보관`/`손질` 헤더로 split → 각 `- ` 리스트로. 헤더 없으면(되묻기 응답) 통째로 한 카드.

### 6-2. 트리거
- **없는 재료 페이지:** `/ingredients/[slug]` 에서 정식 데이터 없음 → 404 대신 발화 카드. (LLM 호출 1회/진입)
- **검색 무결과:** 매칭 재료 0 → "없어요" 대신 발화 카드.

### 6-3. 검색 무분별 호출 방지 (핵심 제약)
LLM은 비용·지연이 있어 **타이핑마다 호출 금지**. 방어 다층:
1. **명시적 트리거만** — 검색은 타이핑 중 자동 호출 X. **엔터/검색 버튼** 등 확정 액션에만.
2. **무결과일 때만** — 매칭 재료가 있으면 호출 안 함. 0건일 때만.
3. **디바운스** — 확정 액션도 짧은 연타 방지(예: 500ms).
4. **캐싱** — 같은 쿼리 재호출 방지(세션/쿼리키 캐시). 동일 재료 재진입 시 재호출 X.
5. **길이·형식 가드** — 너무 짧거나(1글자) 비정상 입력은 호출 전 컷 (또는 API 되묻기에 위임).

### 6-4. 로딩·에러 (API 스펙 기반)
API 계약: `POST /api/ai/chat` → **200**(성공) / **422**(Validation Error). 길이 상한은 API에 없음(2000자도 200) → 프론트가 가드.

**로딩**
- LLM 응답은 수 초 걸린다 → "👩‍🍳 장금이가 생각 중" 인디케이터. (진입 로딩 스켈레톤과 동일 언어)

**에러 케이스별 처리**
| 상황 | 응답/원인 | 화면 |
|---|---|---|
| message 누락·빈값 | **422** `{detail:[{loc,msg}]}` | 호출 전 프론트가 컷(빈 검색·1글자 가드). 그래도 뜨면 "무엇을 여쭤볼지 알려주세요" |
| 정상 무응답성 입력 | **200** 되묻기 answer("어떤 재료가 궁금하신가요?") | 그대로 카드로 노출(에러 아님) |
| 네트워크 실패·타임아웃 | fetch reject / 지연 | "지금은 답을 못 드리겠어요 🥲" + **다시 시도** 버튼 |
| 5xx(서버) | 백엔드 오류 | 위와 동일 재시도 UI |
| 과도한 길이 | 200이지만 낭비 | 프론트가 입력 길이 상한(예: 100자) 가드 후 호출 |

- **폴백 없음** — 없는 재료라 정식 콘텐츠로 폴백할 것이 없다. 에러 시엔 재시도 UI만.
- 검색 무결과에서 온 경우, 에러 시 원래 "조건에 맞는 재료가 없어요"로 우아하게 강등(발화 실패해도 검색 자체는 유지).

## 7. 데이터 & 연동

- **API:** `POST /api/ai/chat` `{ message: string }` → `{ answer: string }`
- **message 구성:** 재료 진입이면 재료명(slug→이름), 검색이면 사용자 입력 원문.
- **answer:** 마크다운형 텍스트. 프론트가 섹션 파싱(정합 스키마 아님 — 자유 텍스트).
- 정식 재료 데이터(zod)와 **별개** — 이 카드는 스키마 대신 answer 텍스트만 씀.

## 8. 성공 지표 (How measure)

- 없는 재료/무결과 진입 후 **이탈률 감소** (빈 화면 대비).
- 발화 카드에서 정식 재료 추천 링크 **클릭률**.
- LLM 호출 수/검색 세션 (무분별 호출이 억제됐는지 — 낮을수록 좋음).
- (후속) 발화 노출 상위 재료 = **정식 콘텐츠 우선순위** 신호.

## 9. 범위 밖 / 미정

- **연속 대화(채팅 UI):** 이번엔 단발 발화만. 대화형은 후속.
- **발화 저장·캐싱 백엔드:** 프론트 세션 캐시만. 서버 캐싱은 후속.
- **AI 답변 품질 검수·필터링:** 부정확·부적절 답변 처리 방침 팀 논의 필요.
- **정식 콘텐츠 전환 트리거:** 발화 자주 뜨는 재료를 언제 정식화할지 기준 미정.
- **비용 관리:** LLM 호출 상한·레이트리밋 정책 (백엔드와 협의).
