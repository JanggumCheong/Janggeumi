# content — 식재료 데이터 콘텐츠 생성 스킬

재료 하나에 대해 사용자가 궁금해하는 정보를, **현재 스키마(`schema.md`)에 맞춰 `examples/<id>.json` 데이터로 채우는** 스킬. 기획(product)이 아니라 **데이터 생성·검증**이 일이다.

## product 스킬과의 경계 (중요)

이 둘을 섞지 않는다. 요청이 어느 쪽인지 먼저 판단한다.

| | **product (기획)** | **content (이 스킬)** |
|---|---|---|
| 입력 | 만들 기능/화면 | 재료 + 사용자 궁금증 |
| 출력 | `docs/plans/*.md` + `*.html` | `examples/<id>.json` |
| 하는 일 | "무엇을 왜, 어떤 필드로" 결정 | 그 필드에 **실제 값**을 채우고 검증 |
| 판단 | 기획적 결정을 내림 | **결정을 내리지 않음** — 근거 있는 사실만 옮김 |

- "감자 콘텐츠 만들어줘", "이 재료 데이터 채워줘", "○○ 궁금증 반영해줘" → **content**.
- "이 기능 기획해줘", "PRD", "화면 어떻게" → **product**.
- 애매하면 사용자에게 어느 쪽인지 묻는다. 기획 문서를 원한 게 아니면 md/html을 만들지 않는다.

## 이 스킬이 지키는 원칙

1. **데이터 생성이지 기획이 아니다.** 필터 축을 새로 설계하거나, 카드의 추천 여부를 판단하거나, 화면 흐름을 바꾸지 않는다. 그건 product/design의 몫이다. 스키마가 이미 정한 형태에 **값만** 넣는다.
2. **근거 없는 값은 넣지 않는다.** [[janggeumi-data-from-image-only]] — 이미지·문서·공식 출처에 없는 값은 추측으로 만들지 않는다. 특히 **분리배출·독성·영양처럼 정답형 정보는 공식 출처(식약처·환경부·농진청)만.** 근거가 없으면 그 필드를 **비운다**(빈 필드는 UI 미노출). "일단 그럴듯하게"가 가장 위험하다.
3. **임의 수치를 지어내지 않는다.** `rating`(avg/count/dist)은 **후속 UGC 전까지 목업 임시값**이다(→ [[design-notes]] "별점은 목업 임시값"). content 작업에서 rating에 **의미를 부여하지 않는다** — "비추천이니 낮게" 같은 판단은 기획적 개입이라 금지. 기존 재료 관례(4.x대)를 따르거나, 값의 근거가 없으면 rating을 **생략**한다. avg를 낮춰 "이 방법은 나쁘다"를 표현하려 하지 말 것 — 그 신호는 `recommended:false`·`cautions`·`notFor`가 담당한다.
4. **스키마가 진실의 원천.** 필드명·구조는 `schema.md`를 그대로 따른다. 새 필드가 필요하면 임의 추가 말고 사용자에게 확인 → schema.md에 먼저 반영.
5. **작업 후 반드시 검증(아래 6장).** 정합성이 곧 데이터 품질이다.

## 작업 순서

### 0. 궁금증 발굴 (필수 — 이 스킬의 출발점)

이 스킬은 "**사용자가 궁금해할 만한** 정보"를 만든다. 사용자가 궁금증 목록을 주지 않았어도(혹은 몇 개만 줬어도), **내가 먼저 발굴**한다. 여기서 좁게 시작하면 이후 전부가 그 프레임에 갇힌다 — 넓게 연다.

- **발굴 소스:** 웹 검색으로 그 재료의 **연관검색어·자주 묻는 질문(FAQ)·"○○ 왜"·계절/상황 이슈**를 모은다. 실제 사람들이 검색하는 형태로.
- **세 영역에 고루:** 구매(고르기)·보관(어디/얼마)·처리(활용/폐기)에 각각 궁금증이 있는지 훑는다. 한 영역에 쏠리지 않게.
- **후보를 사용자에게 제시하고 범위를 확정받는다.** 발굴한 궁금증을 나열하고 "이 중 무엇을 데이터로 담을지" 사용자가 고르게 한다. **내가 임의로 3개로 좁히지 않는다.** (이게 이전 실패의 핵심 원인.)
- 확정된 궁금증만 이후 단계로 넘어간다.

### 1. 대상 재료·현 상태 확인
- `list.json`에서 재료가 등록돼 있는지, `status`(published/draft/planned)를 본다.
- **이미 있는 재료면** `examples/<id>.json`을 읽고, 확정한 궁금증이 **이미 담겼는지 / 얕게만 담겼는지 / 없는지** 대조한다. 없는 재료면 `schema.md` 최상위 구조부터 새로 만든다.

### 2. 뿌리 문서 읽기 (필수)
- `schema.md` — 필드 규약(구매·보관·처리 구조, source 객체, 검증 규칙). **이게 계약이다.**
- `design-notes.md` — 구매=정답형/보관=합의형/처리=확산형. 궁금증이 어느 영역인지, 그 영역의 신뢰 모델(출처 무게)을 잡는다.

### 3. 궁금증 → 필드 매핑
사용자 궁금증을 스키마 필드에 1:1로 건다. 재료명·필터를 하드코딩하지 않는다.

| 궁금증 유형 | 영역 | 담기는 필드 |
|---|---|---|
| 좋은 거 고르는 법 | purchase | `criteria[]`(+`source` 필수), `bestWorst` |
| 어디에·어떻게 보관 | storage | `methods[]`(tags=filters축과 1:1, `source` 필수), `tip` |
| 왜 이 보관이 나쁜가 | storage | 해당 method `cautions`·`notFor` (rating 아님) |
| 남은 걸로 뭘 만드나 | handling | `recipes[]` |
| 어떻게 버리나 / 왜 | handling | `dispose[]`: `way`(요약) + 상세 `reason`·`regionNote`·`cautions`·`source` |
| 왜 이런 현상이 나나 (싹·갈변 등) | 해당 영역 `cautions`/`reason` 또는 `related` | 지식 설명 |

### 4. 근거 확보 (딥링크 의무)
정답형 정보(분리배출·독성·보관 과학)는 **공식 출처를 웹 검색으로 확보**해 `source`에 기관·type·url·lastReviewed를 채운다. 근거를 못 찾으면 그 필드를 비우고 사용자에게 알린다. 지어내지 않는다.

- **`source.url`은 딥링크여야 한다.** 홈 도메인(`https://www.mfds.go.kr`)이 아니라 **그 주장이 실제로 있는 구체 페이지 URL**이어야 한다. 홈 URL은 "거기 어딘가 있겠지"라 교차확인이 불가능하다 — 검증 스크립트가 홈/루트 URL을 **불합격**시킨다.
- 딥링크를 못 찾으면: (a) 다른 공식 출처를 더 찾거나 (b) 그 주장을 빼거나 (c) `verified:false`로 낮추고 사용자에게 보고. 홈 URL로 `verified:true`를 붙이지 않는다.
- 확보한 딥링크는 `WebFetch`로 **실제 그 페이지에 그 주장이 있는지** 확인한 뒤 채운다.

### 5. 값 채우기
`schema.md`의 구조 그대로. 상세 필드(dispose의 `reason`/`steps`/`cautions`/`regionNote`/`source`/`detailHref`)는 **근거 있는 것만**. 화자 코멘트는 `author.type:"curator"`(장금이).

### 6. 검증 (필수 — 스킬의 핵심)

**이 스킬의 존재 이유는 "신뢰할 수 있는 데이터"다.** 검증을 두 층으로 한다. **6-A(정보 신뢰성)가 우선**이고, 6-B(형식 정합성)는 기계적 최소 요건이다. 형식만 통과하고 사실이 틀린 데이터가 가장 위험하다.

#### 6-A. 정보 신뢰성 검증 (사실이 맞는가 — 스킬의 심장)

값 하나하나가 **참인가**를 검증한다. 정합성이 아니라 **사실**이다.

1. **모든 사실 주장에 출처가 있는가.** 구매 기준·보관 과학·독성·분리배출·영양 수치 등 **정답형 주장**은 근거가 있어야 한다. 근거 없는 주장은 넣지 않는다(비운다). "그럴듯함"은 근거가 아니다.
2. **출처 등급이 주장의 무게에 맞는가.** → `schema.md`의 source `type` 등급.
   - **건강·안전·독성·분리배출** = 반드시 `official`(식약처·환경부·농진청 등). `media`(유튜버·블로그)만으로 이런 주장을 실으면 **불합격** — `verified:false` 처리하고 사용자에게 경고.
   - 일반 보관·손질 팁 = `official`/`expert` 권장, `media`는 `media[]`(참고)로만.
3. **교차 확인(cross-check).** 중요한 정답형 주장(특히 안전·분리배출)은 **최소 2개 독립 출처**가 일치하는지 웹 검색으로 확인한다. 한 출처만 있으면 그 사실을 그대로 단정하지 말고 톤을 낮추거나(‘~일 수 있어요’) `regionNote`로 불확실성을 명시한다.
4. **수치·기준의 출처 일치.** 온도(`conditions.temp`)·기간(`durationDays`)·칼로리·영양(`facts`)은 **출처 값과 일치**해야 한다. 임의 반올림·창작 금지.
5. **지역·조건 의존성 표시.** 분리배출처럼 지자체·상황에 따라 답이 갈리는 건 단정하지 말고 `regionNote`로 "관할 확인"을 명시(정답형의 함정).
6. **rating 등 UGC 수치는 사실 주장이 아니다.** 후속 UGC 전 목업 임시값이므로 **의미를 부여하지 않는다**(원칙 3). 검증 대상은 "합이 맞나"(형식)일 뿐, avg의 높낮이에 서사를 넣지 않는다.
7. **최신성.** 각 `source.lastReviewed`가 오래됐으면(기준이 바뀌는 분리배출 등) 재확인. 스킬 실행 시점으로 갱신.
8. **추측 필드 색출.** 이미지·문서·출처 어디에도 없는데 "채워두면 좋을 것 같아" 넣은 값이 있는지 되짚어 제거한다. → [[janggeumi-data-from-image-only]].

> **불합격 처리:** 6-A를 통과 못 한 주장은 (a) 근거를 찾아 채우거나 (b) 필드를 비우거나 (c) 톤을 낮춘다(단정 → 조건부). **틀릴 바엔 비운다.** 검증 결과(무엇을 어떤 출처로 확인했는지, 무엇을 못 채웠는지)를 마무리에 보고한다.

#### 6-B. 형식·출처 정합성 검증 (자동화 — `scripts/validate-content.mjs`)

산문 규칙은 안 지켜도 통과된다. **자동으로 걸러지는 것은 코드로 강제**한다. 값을 채운 뒤 반드시 실행:

```
node .claude/skills/content/scripts/validate-content.mjs <id>
```

스크립트가 검사하는 것(불합격 시 exit 1 + 어디가 왜 틀렸는지 리포트):

**형식 정합성**
- **JSON 유효** (parse OK)
- `purchase.criteria[].source`·`storage.methods[].source` **필수**(존재)
- `storage.methods[].tags`의 각 key/value가 `storage.filters[]`에 **존재**. **필터 옵션마다 매칭 카드 ≥ 1**(빈 필터 금지).
- `rating`이 **있으면** `dist` 합 ≈ 100 (rating 자체는 선택 — 원칙 3)
- `dispose[]` 있으면 `key/title/way/wasteType` 필수, `wasteType ∈ {food,general,recycle}`
- 모든 `id`/`key`는 영문 slug, 재료 내 유일

**출처 신뢰성(6-A의 자동화 가능 부분)**
- 모든 `source.url`이 **URL 형식**이고 **딥링크**인가 — 루트/홈(`https://www.mfds.go.kr`, `.../` 로 끝나는 경로 없는 URL)이면 **불합격**. `verified:true`인데 `url`이 없어도 **불합격**(근거 없이 검증 주장 금지). `verified:false` + `url` 없음은 "딥링크 미확보로 정직하게 낮춤"이라 허용하되 경고.
- `source.type`이 `official`인데 URL 호스트가 **공식 기관 화이트리스트**(`mfds.go.kr`, `me.go.kr`, `rda.go.kr`, `nongsaro.go.kr`, `korea.kr` 등)에 없으면 **경고**(official 참칭 방지).
- 건강·안전·독성·분리배출 맥락(`dispose[].wasteType`, 독성 관련 `criteria`/`cautions`) 항목이 `source` 없이 단정하면 **경고**.
- `source.verified:true`인데 `type`이 `media`/`community`면 **불합격**(schema 규칙: official/expert만 verified).

> 스크립트가 없거나 재료에 새 검사 패턴이 필요하면 이 파일과 함께 스크립트를 업데이트한다. **자동으로 못 잡는 것(주장이 실제로 참인가, 딥링크에 그 내용이 있는가)은 6-A대로 사람이 `WebFetch`로 확인한다** — 스크립트는 최소 방어선이지 사실 검증의 대체가 아니다.

## 마무리
- 채운 파일 경로와 **무엇을 어떤 필드에 넣었는지** 요약.
- 근거를 못 채워 비운 필드가 있으면 명시.
- **기획적 결정이 필요한 지점(새 필터 축, 추천 여부, 새 화면)이 나오면 여기서 멈추고 product/design 스킬로 넘긴다.** content가 임의로 정하지 않는다.
