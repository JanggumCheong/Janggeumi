# product — 기능 기획서 생성 스킬

장금이 프로젝트의 **세부 기능 기획서**를 반복 생성한다. 결과물은 항상 두 가지: 기획 markdown + 시각화 HTML.

## 0. 대상 기능

기획할 기능: **$ARGUMENTS**

인자가 비어 있으면 사용자에게 "어떤 기능을 기획할까요?"를 먼저 묻는다.

## 1. 뿌리 문서 먼저 읽기 (필수)

기획을 시작하기 전에 **반드시** 프로젝트 뿌리 문서를 읽고 방향을 맞춘다:

- `jonggil/foundation.md` — 목적/타겟/핵심가치/차별점/톤. **세부 결정이 이 파일과 어긋나면 이 파일이 우선한다.**

`foundation.md`가 없으면 사용자에게 알리고, 뿌리 문서 없이 진행할지 확인한다.

추가로 관련 스프린트 초안(`jonggil/04_*`, `05_*`, `06_*` 등)이 있으면 참고한다. 최신 팀 논의가 필요하면 사용자에게 FigJam 보드 반영 여부를 묻는다(Figma MCP 한도가 있을 수 있음).

## 2. 기획서 작성 원칙

- **foundation.md의 핵심 가치를 기능에 매핑한다.** 각 기능 결정이 "빠른 확신 / 믿음(집단지성) / 끝까지(안 버리기)" 중 무엇을 지지하는지 명시한다.
- **구매·보관·처리는 성격이 다르다.** 기능이 어느 영역인지 파악하고 그 영역의 정보 성격(구매=정답형·읽기 / 보관=합의형·필터+평가 / 처리=확산형·UGC)에 맞춘다. `design-notes.md` 참조.
- **동어반복·모호함 금지.** "쉽게 만든다"가 아니라 "무엇을 어떻게" 구체적으로.
- 단정 톤. 배경·이유 장황하게 늘어놓지 말고 결정과 명세를 적는다.

## 3. 기획서 구조 (markdown)

`jonggil/<기능명>-기획.md`로 저장. 아래 섹션을 채운다:

1. **한 줄 정의** — 이 기능이 무엇인지 한 문장.
2. **문제 (Why)** — 어떤 pain을 푸는가. foundation의 pain과 연결.
3. **대상 & 시나리오 (Who/When)** — 누가, 어떤 상황에서 쓰나. 대표 시나리오 1~2개(구체적 흐름).
4. **핵심 가치 매핑** — 이 기능이 지지하는 foundation 핵심가치(빠른 확신/믿음/끝까지).
5. **영역 & 정보 성격** — 구매/보관/처리 중 무엇이며 그에 따른 UI·데이터 방향.
6. **기능 명세** — 화면·동작 목록. 각 항목은 사용자 행동 → 시스템 반응 형태로.
7. **데이터 & 필터** — 필요한 데이터 항목. (보관이면 재료 타입별로 필터가 달라진다는 점 반영.)
8. **성공 지표 (How measure)** — 이 기능이 성공했다고 볼 측정 가능한 신호.
9. **범위 밖 / 미정** — 이번에 안 하는 것, 팀과 정할 것.

## 4. 시각화 HTML 생성 (필수)

기획 내용을 팀이 한눈에 보도록 HTML로 시각화한다. `jonggil/<기능명>-기획.html`로 저장.

- 템플릿 [template.html](template.html)의 구조·스타일을 기반으로 만든다. 절대 매번 처음부터 스타일을 지어내지 말 것 — 템플릿 톤(장금이 색·카드·위계)을 유지해 문서 간 일관성을 지킨다.
- 담을 것: 한 줄 정의 히어로 · 문제/대상 카드 · 핵심가치 매핑 배지 · 기능 명세 리스트 · 성공지표 · 미정 항목.
- 기능이 특정 화면을 다루면, `design` 스킬의 목업과 연결(링크)하거나 간단한 와이어프레임 블록을 넣는다. 실제 화면 목업이 필요하면 `design` 스킬을 안내한다.

## 5. 마무리

- 저장한 두 파일 경로를 사용자에게 알린다.
- foundation.md의 "미정" 항목과 충돌하거나 새로 정해야 할 결정이 나오면 명시적으로 질문한다.
