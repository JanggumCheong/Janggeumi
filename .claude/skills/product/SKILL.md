# product — 기능 기획서 생성 스킬

장금이 프로젝트의 **세부 기능 기획서**를 반복 생성한다. 결과물은 항상 두 가지: 기획 markdown + 시각화 HTML.

## 0. 대상 기능

기획할 기능: **$ARGUMENTS**

인자가 비어 있으면 사용자에게 "어떤 기능을 기획할까요?"를 먼저 묻는다.

## 1. 뿌리 문서 먼저 읽기 (필수)

기획을 시작하기 전에 **반드시** 프로젝트 뿌리 문서를 읽고 방향을 맞춘다:

- `docs/foundation.md` — 목적/타겟/핵심가치/차별점/톤. **세부 결정이 이 파일과 어긋나면 이 파일이 우선한다.**

`docs/foundation.md`가 없으면 사용자에게 알리고, 뿌리 문서 없이 진행할지 확인한다.

추가로 관련 스프린트 초안(`docs/` 하위)이 있으면 참고한다. 최신 팀 논의가 필요하면 사용자에게 FigJam 보드 반영 여부를 묻는다(Figma MCP 한도가 있을 수 있음).

## 2. 기획서 작성 원칙

- **foundation.md의 핵심 가치를 기능에 매핑한다.** 각 기능 결정이 "빠른 확신 / 믿음(집단지성) / 끝까지(안 버리기)" 중 무엇을 지지하는지 명시한다.
- **구매·보관·처리는 성격이 다르다.** 기능이 어느 영역인지 파악하고 그 영역의 정보 성격(구매=정답형·읽기 / 보관=합의형·필터+평가 / 처리=확산형·UGC)에 맞춘다. `design-notes.md` 참조.
- **데이터 정합성이 곧 기획이다.** 화면의 필터·필드·카피는 실제 데이터 스키마(`content/schema.md`, 재료별 JSON)와 **1:1로 맞아야 한다.** 데이터에 없는 필터('선호도' 같은 것)를 화면에 넣지 않는다. 필터는 그 재료의 `storage.filters` 축과 일치, 방법은 각 `tags`로 걸러진다. 재료명을 UI에 하드코딩하지 않는다(양파·계란도 같은 화면을 쓴다).
- **SEO/GEO를 기획 단계에서 정한다.** 검색 진입·AI 인용이 성장 동력이다. 화면마다 h1(검색어)·메타·구조화 데이터(JSON-LD, 출처는 `citation`)를 기획에 포함. 접는 콘텐츠(출처 등)도 SSR로 DOM에 존재시켜 색인되게. → `schema.md`의 신뢰 모델(출처 등급)과 연결.
- **장금이 톤 일관.** 화면 카피는 "👩‍🍳 장금이 왈, ~것이옵니다" 화자를 세 영역 공통으로. 사극 톤은 카피에서만 가볍게(디자인은 현대적).
- **동어반복·모호함 금지.** "쉽게 만든다"가 아니라 "무엇을 어떻게" 구체적으로.
- 단정 톤. 배경·이유 장황하게 늘어놓지 말고 결정과 명세를 적는다.

## 2-0. API ↔ 프론트 스키마 정합 (백엔드 계약이 생겼을 때)

프론트 zod/JSON은 **화면 기획 우선**으로 먼저 만들어졌고, 백엔드 API는 **DB 정규화 우선**이라 필드명·구조가 어긋난다. 실제 API가 나오면 **필드별로 대조**해 아래 4분류로 나누고, **격차 큰 항목은 임의로 바꾸지 말고 사용자 확인 → 백엔드와 논의**한다. **API를 진실의 원천으로 삼되, 무비판 추종하지 않는다(결함은 되짚는다).**

각 필드를 다음으로 분류한다:
1. **매핑** — 네이밍·계층 차이(`purchase_tips`→`criteria`, `content`→`desc`, 최상위→`purchase.criteria` 중첩). **프론트 loader 어댑터가 흡수**한다. 화면 컴포넌트는 프론트 zod 타입에만 의존하므로 어댑터가 API 이질성을 한 곳에서 번역하면 **UI 무변경**이고, API가 흔들려도 어댑터만 고친다. "API가 진실의 원천"은 **값**의 기준일 뿐, **화면이 쓰는 타입 형태**는 프론트 zod가 계약 — 어댑터가 이 둘을 잇는다. (백엔드 응답 구조 변경을 굳이 기다리지 않는다. 단 값·데이터 자체가 없는 누락/깨짐은 어댑터로 못 만드니 백엔드 요청.)
2. **누락(프론트에 있고 API에 없음)** — 화면에 실제로 쓰이면 **백엔드에 추가 요청**(예: `source` 출처 등급, `bestWorst`). 신뢰·SEO 근간 필드는 우선순위 높게.
3. **깨짐(값이 오염/뭉침)** — 예: `title`이 전부 "꿀팁"이고 진짜 제목이 `content`에 `"N. 제목: 설명"`으로 섞임 → **백엔드에서 분리 요청**(프론트 파싱은 깨지기 쉬움, 최후수단).
4. **죽은 데이터(화면·API 둘 다 없음)** — 즉시 **제거**(zod·JSON·`schema.md` 동기화). 예: `purchase.intro`, 과거 `categories`·`contribution`. "이미지·문서에 없는 필드는 추측으로 넣지 말 것", "UI 없는 데이터 없음" 원칙과 같은 결.

판정 전 **화면 컴포넌트가 그 필드를 실제로 렌더하는지 grep으로 검증**한다(다른 이름·하드코딩으로 대체됐을 수 있음). 규칙 생성 가능한 값(예: `headline`=`좋은 {재료} 고르는 법`)은 API 요청 대신 **프론트 생성**으로 뺀다.

정합 논의는 **디스코드에 가독성 있게 공유**한다: 두 스키마 성격 차이 → 매핑/누락/깨짐/죽은데이터 분류 → 정하고 싶은 것. 결정이 나면 `content/schema.md` 규약과 이 스킬에 반영한다.

## 2-1. 논의로 기획을 구체화한다 (필수 워크플로)

기획은 한 번에 완성되지 않는다. **질문 → 결정 → 반영 → 명세 싱크**를 반복한다:
1. 애매한 선택(색·레이아웃·필터 방식·기능 범위)은 **추천안을 제시하고 사용자에게 확정**받는다. 임의로 정하지 않는다.
2. 판단 기준: 기능은 **foundation 핵심 가치**, 화면은 **"토스였어도 이렇게 했을까?"**. 참조 이미지가 있으면 그것이 스펙.
3. 결정이 나면 기획서·시각화에 반영하고, **화면 목업이면 우측 기획 명세 패널(`spec-*`, design-system.md 8-5)과 즉시 싱크**한다. 목업과 명세가 어긋나면 안 된다.
4. 로그인·UGC 등 후속 기능에 의존하는 요소(사용자 후기·평점)는 **지금 채울 수 있는 대체안**(예: 장금이 코멘트=검증된 큐레이션)으로 기획하고, 후속임을 명시한다.

## 3. 기획서 구조 (markdown)

`docs/plans/<기능명>-기획.md`로 저장. 아래 섹션을 채운다:

1. **한 줄 정의** — 이 기능이 무엇인지 한 문장.
2. **문제 (Why)** — 어떤 pain을 푸는가. foundation의 pain과 연결.
3. **대상 & 시나리오 (Who/When)** — 누가, 어떤 상황에서 쓰나. 대표 시나리오 1~2개(구체적 흐름).
4. **핵심 가치 매핑** — 이 기능이 지지하는 foundation 핵심가치(빠른 확신/믿음/끝까지).
5. **영역 & 정보 성격** — 구매/보관/처리 중 무엇이며 그에 따른 UI·데이터 방향.
6. **기능 명세** — 화면·동작 목록. 각 항목은 사용자 행동 → 시스템 반응 형태로.
7. **데이터 & 필터** — 필요한 데이터 항목. (보관이면 재료 타입별로 필터가 달라진다는 점 반영.)
8. **성공 지표 (How measure)** — 이 기능이 성공했다고 볼 측정 가능한 신호.
9. **범위 밖 / 미정** — 이번에 안 하는 것, 팀과 정할 것.

## 4. 시각화 HTML 생성 (필수)

기획 내용을 팀이 한눈에 보도록 HTML로 시각화한다. `docs/plans/<기능명>-기획.html`로 저장.

- 템플릿 [template.html](template.html)의 구조·스타일을 기반으로 만든다. 절대 매번 처음부터 스타일을 지어내지 말 것 — 템플릿 톤(장금이 색·카드·위계)을 유지해 문서 간 일관성을 지킨다.
- 담을 것: 한 줄 정의 히어로 · 문제/대상 카드 · 핵심가치 매핑 배지 · 기능 명세 리스트 · 성공지표 · 미정 항목.
- 기능이 특정 화면을 다루면, `design` 스킬의 목업과 연결(링크)하거나 간단한 와이어프레임 블록을 넣는다. 실제 화면 목업이 필요하면 `design` 스킬을 안내한다.
- **화면 목업엔 우측 기획 명세 패널을 함께 둔다.** 목업(폰 화면) 옆에 데이터 매핑(어떤 schema 필드를 쓰는지)·라우팅·탭별 기능 요구사항·확정 사항을 표로 나란히 보인다(design-system.md 8-5의 `spec-*`). 기획서(md)의 명세와 목업 패널이 같은 내용을 가리키게 유지한다. 완성 예시: [`docs/mockups/ingredient-detail-mockup.html`](../../../docs/mockups/ingredient-detail-mockup.html).

## 5. 마무리

- 저장한 두 파일 경로를 사용자에게 알린다.
- foundation.md의 "미정" 항목과 충돌하거나 새로 정해야 할 결정이 나오면 명시적으로 질문한다.
