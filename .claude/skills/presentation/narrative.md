# 피칭 서사 · 대본 레퍼런스

> `presentation` 스킬의 기준 문서. 슬라이드 흐름·슬라이드별 내용·대본 톤을 여기서 정한다.
> 값(문제·차별점·지표)은 프로젝트 실제 상태에서 가져온다. 근거 없는 숫자·기능은 넣지 않는다(→ [[janggeumi-data-from-image-only]] 원칙과 같은 결).

---

## 0. 청중 = 투자자 / 심사 피칭

- 목적은 **설득과 임팩트**다. 기능 나열이 아니라 "왜 이 문제가 크고, 왜 우리가 이기나"를 남긴다.
- 한 슬라이드 = 한 메시지. 발표자가 말하고, 슬라이드는 **근거·이미지**만 받친다. 슬라이드에 문장을 다 쓰지 않는다.
- 마지막은 감사 인사가 아니라 **요청(Ask)**으로 끝낸다.

## 1. 기본 서사 = 문제→해결 고전 구조 (YC·시퀀이아 계열)

기본 슬라이드 순서. 발표 길이·목적에 따라 늘리거나 줄이되 **순서는 유지**한다.

| # | 슬라이드 | 한 메시지 | 근거로 넣을 것 |
|---|---------|----------|---------------|
| 1 | **표지 / 훅** | 한 문장으로 무엇을 없애는가 | 태그라인 + 마스코트(가볍게) |
| 2 | **문제** | 식재료는 사는 순간부터 버리는 순간까지 고민 | 세 국면(구매·보관·처리)의 반복 고민, 가능하면 실측 지표 |
| 3 | **기존 방식의 한계 / Why now** | 왜 지금까지 부족했나 | 지금 vs 장금이 대비 (흩어짐·신뢰불가·따로놈) |
| 4 | **해결 (제품 한 줄)** | 재료 하나 검색 → 사고·두고·처리 끝 | 3영역 컬럼 + 장금이 화자 한 줄 |
| 5 | **데모** | 말 대신 화면 | **실제 앱 캡처**(Playwright). 라이브 시연 권장 |
| 6 | **차별점 / Moat** | 검색·AI가 우리를 인용하게 설계 | SEO·GEO·출처 등급. 마케팅비 없이 크는 구조 |
| 7 | **다음 / 요청 (Ask)** | 지금까지 → 다음, 그리고 요청 | 완료/진행/다음 3단 + 한 문장 Ask |

**확장 옵션** (시간·목적 맞춰 삽입):
- 시장 규모(TAM/SAM/SOM) — 문제와 해결 사이. 실측 근거 있을 때만.
- 비즈니스 모델 — Moat 뒤. 아직 미정이면 "가설"로 명시.
- 팀 — Ask 직전. 왜 이 팀이 이기는가.
- 경쟁 지형 — Moat와 함께. 2x2 또는 표.

## 2. 슬라이드별 작성 원칙

- ⚠️ **발표 산출물(슬라이드 노출 카피·발표 대본)에 특정 회사 브랜드명을 넣지 않는다.** "토스" 등은 내부 디자인 판단 기준(design-system.md)일 뿐, 발표에서 인용하지 않는다. 간결·직관·현대는 **장금이 자신의 언어**로 표현한다. (내부 노트에서 우리끼리 기준으로 언급하는 것과, 청중에게 남 브랜드를 파는 것은 다르다.)
- **훅(1번)은 고민의 반복성을 찌른다.** "무엇을 살까"가 아니라 "사고·두고·처리가 다 고민"이라는 우리 고유 프레임. → [[janggeumi-project-topic]] ("제철" 아님, 구매·보관·처리).
- **문제(2번)에 근거 없는 숫자를 넣지 않는다.** 실측치가 있으면 metric으로, 없으면 정성적 세 국면으로. 물음표(`?`) 카드로 "확신 없음"을 시각화하는 건 OK.
- **해결(4번)은 세 영역이 성격이 다름을 보여준다.** 구매=정답형 / 보관=합의형 / 처리=확산형. 같은 카드 3개 금지. → design-notes 성격 구분과 일치.
- **데모(5번)가 피칭의 심장.** 실제 화면 캡처. 가능하면 라이브. "Show, don't tell."
- **기존 한계(3번)는 "흩어짐·신뢰불가·따로놈" 세 축으로 대비한다.** 다른 서비스 브랜드명을 끌어오지 말고 현상으로 서술한다.
- **차별점(6번)은 SEO/GEO를 성장 엔진으로 건다.** 화면마다 h1·JSON-LD·출처 등급 → 검색·AI 인용. 신뢰 모델(출처 등급)이 근간.
- **Ask(7번)는 발표 목적에 맞는 한 문장.** 투자·협업·채용·피드백 중 하나로 명확히. 감사보다 다음 행동.

## 3. 장금이 톤 (피칭에서도 가볍게)

- 마스코트·한 줄 멘트 수준으로만 "👩‍🍳 장금이 왈, ~것이옵니다" 화자를 쓴다. 슬라이드 전체를 사극 톤으로 덮지 않는다.
- 디자인은 **모던·미니멀**(→ 세이지 그린 #5C6E3D 포인트, [[janggeumi-brand-color]]). 전통/한지/붓글씨 금지. 간결함은 우리 화면·카피로 증명하지, 남 브랜드 이름을 빌리지 않는다.
- 발표 대본(구어)은 사극 톤이 아니라 **자연스러운 존댓말**. 화자 멘트만 슬라이드에서 위트로.

## 4. 발표자 노트(대본) 작성 규칙

각 슬라이드 `<aside class="notes">`에 그 슬라이드에서 **말할 것**을 적는다(화면엔 안 보임, `S` 키로 토글).

- 슬라이드 텍스트를 읽지 말고 **보완**한다. 슬라이드=근거, 대본=서사.
- 한 슬라이드 대본 = 말로 **20~40초** 분량(2~4문장). 길면 슬라이드를 쪼갠다.
- 전환 문장을 넣는다("그래서 우리가 한 게…", "이걸 실제로 보면…").
- 숫자·주장은 **출처를 아는 것만**. 모르면 대본에 `[근거 확인]` 표시를 남겨 사용자에게 묻는다.

## 5. UI/UX 상호작용 (덱 자체)

- 좌우 화살표 / Space / 클릭 = 다음, `←`·`PageUp` = 이전.
- `F` 전체화면, `S` 발표자 노트, `O` 개요(그리드), `Esc` 개요 닫기.
- `#3` 딥링크로 특정 슬라이드 진입, 진행바로 위치 표시.
- 인쇄 → 슬라이드당 한 페이지(PDF 내보내기). 외부 리소스 로드 없음(오프라인 발표 안전).
