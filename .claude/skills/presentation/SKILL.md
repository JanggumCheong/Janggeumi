# presentation — 피칭 덱 생성 스킬

장금이 프로젝트의 **발표 자료**를 만든다. 결과물은 자체 완결형 HTML 슬라이드 덱 하나(+ 필요 시 실제 앱 캡처). 매번 스타일을 새로 짓지 않고 디자인 토큰·서사 구조를 재사용한다.

## 0. 대상 발표

만들 발표: **$ARGUMENTS**

인자가 비어 있으면 "무슨 발표를 만들까요? (청중·목적·발표 시간)"를 먼저 묻는다.

## 1. 기준 문서 먼저 읽기 (필수)

덱을 만들기 전에 **반드시** 읽는다:

1. [narrative.md](narrative.md) — 피칭 서사 구조·슬라이드별 내용·대본 톤·UI 규칙. **여기 정의된 흐름과 규칙을 따른다.**
2. `.claude/skills/design/design-system.md` — 색·타이포·간격 토큰. **덱은 이 토큰만 쓴다.** 임의 색 금지.
3. `docs/` 하위 관련 기획·목업(`docs/plans/`, `docs/mockups/`) — 발표에 넣을 실제 내용·화면의 원천.

프로젝트 방향(목적/타겟/핵심가치)은 위 문서와 메모리에서 가져온다. 근거 없는 주장·숫자·기능은 넣지 않는다.

## 2. 절대 원칙

- ⚠️ **발표 산출물에 특정 회사 브랜드명을 넣지 않는다.** "토스" 등은 내부 디자인 판단 기준(design-system.md)일 뿐, **슬라이드 노출 카피·발표 대본에서 인용하지 않는다.** 간결·직관·현대는 **장금이 자신의 언어**로 표현한다. (내부에서 기준으로 삼는 것과, 청중에게 남 브랜드를 파는 것은 다르다.)
- **현대적으로.** 장금이는 매개체일 뿐. 전통/사극/한지/붓글씨 시각 요소 금지. 세이지 그린 #5C6E3D 포인트. "요즘 잘 만든 발표처럼 보이나?"가 기준.
- **토큰만 사용.** 색·간격·반경은 design-system.md 값에서만. 덱 템플릿의 CSS 변수를 그대로 쓴다.
- **한 슬라이드 한 메시지.** 슬라이드는 근거·이미지, 발표자가 서사를 말한다. 슬라이드에 문장을 다 쓰지 않는다.
- **데이터에 없는 내용 금지.** 숫자·지표·기능은 실제 프로젝트 상태에서만. 모르면 대본에 `[근거 확인]`을 남겨 사용자에게 묻는다.
- **논의로 구체화.** 애매한 선택(강조할 지표·슬라이드 추가·Ask 문구)은 추천안 제시 후 사용자 확정.

## 3. 서사 = 문제→해결 고전 구조 (기본)

기본 흐름(청중=투자자·심사 피칭). 상세는 [narrative.md](narrative.md) 1장 표.

`표지/훅 → 문제 → 기존 한계(Why now) → 해결(제품 한 줄) → 데모 → 차별점(Moat) → 다음/요청(Ask)`

- 발표 시간·목적에 맞춰 확장 슬라이드(시장·BM·팀·경쟁)를 삽입하되 순서는 유지.
- 세 영역(구매=정답형 / 보관=합의형 / 처리=확산형)이 성격이 다름을 해결 슬라이드에서 보여준다. 같은 카드 3개 금지.
- 마지막은 감사 인사가 아니라 **Ask**로 끝낸다.

## 4. 덱 생성

1. [deck-template.html](deck-template.html)을 `docs/presentations/<주제>-deck.html`로 복사한다. (`docs/presentations/`가 없으면 만든다.)
2. `<section class="slide">`의 **내용만** 채운다. `<style>`의 토큰·컴포넌트 클래스(`.metrics`·`.areas`·`.compare`·`.shots`·`.quote` 등)는 그대로 재사용.
3. 슬라이드 수가 바뀌면 표지 슬라이드에만 `active`를 남기고, 하단 `#total`은 스크립트가 자동 계산하므로 손대지 않는다.
4. 각 슬라이드 `<aside class="notes">`에 **발표 대본**을 채운다(narrative.md 4장 규칙: 한 슬라이드 20~40초, 슬라이드를 읽지 말고 보완, 전환 문장 포함).
5. 마스코트·이미지는 **상대경로 또는 data URI**로 넣어 덱을 자체 완결형으로 유지한다(오프라인 발표 안전). 캐릭터 원본: `.claude/skills/design/Janggeumi-character.png`.

### 덱 조작 (발표자용)
좌우 화살표/Space/클릭=다음, `←`=이전, `F`=전체화면, `S`=발표자 노트, `O`=개요 그리드, `#3`=딥링크, 인쇄=슬라이드당 1페이지(PDF).

## 5. 데모 슬라이드 = 실제 앱 캡처 (자동화)

데모 슬라이드는 **실제 화면**으로 채운다("Show, don't tell"). 스크래치패드에 Playwright를 격리 설치하고 주요 화면을 캡처한다:

1. `npm run dev`로 앱을 띄운다(포트 확인).
2. Playwright로 주요 라우트를 캡처: `/`(홈), `/search`, `/ingredients/[slug]`, `/ingredients/[slug]/[tab]`(구매·보관·처리 각 탭). 모바일 뷰포트(≈390px)로 찍어 폰 프레임(`.phone-shot`)에 넣는다.
3. 캡처 PNG를 `docs/presentations/shots/`에 저장하고 덱에서 상대경로로 참조.
4. **앱이 안 뜨거나 화면이 비면** `docs/mockups/`의 확정 목업 HTML을 캡처해 fallback으로 쓴다(design 스킬이 만든 검증된 시안). 어느 쪽을 썼는지 사용자에게 알린다.

## 6. 검증

- **Playwright로 덱 렌더를 검증**한다(스크래치패드 격리 설치). 각 슬라이드를 풀스크린(16:9)으로 스크린샷 찍어 **넘침·겹침·잘림·타이포 스케일**을 실제로 확인한다. cqw 단위라 뷰포트별로 다르게 보일 수 있으니 발표 해상도(예: 1920×1080)로 검증.
- 발표자 노트(`S`)·개요(`O`)·진행바가 동작하는지 확인.

## 7. 논의로 구체화 (필수 워크플로)

발표는 한 번에 완성되지 않는다. **질문 → 결정 → 반영 → 검증**을 반복한다:
1. 강조할 지표·추가 슬라이드·Ask 문구·데모 범위 등 애매한 선택은 **추천안을 제시하고 사용자 확정**. 임의로 정하지 않는다.
2. 근거 없는 숫자·기능은 넣지 말고 `[근거 확인]`으로 남겨 묻는다.
3. 결정이 나면 덱·대본에 반영하고 Playwright로 재검증.

## 8. 마무리

- 저장한 덱 경로·캡처 경로·슬라이드 수를 알린다.
- narrative.md/design-system.md에 없어서 새로 정한 패턴·컴포넌트가 있으면 명시하고, 반영할지 사용자에게 확인한다.
